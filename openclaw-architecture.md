## 一、产品定位

**OpenClaw = 自托管的多渠道AI Agent网关**

| 特性 | 说明 |
|------|------|
| 核心定位 | 自托管的AI Agent网关，连接聊天应用到AI模型 |
| 目标用户 | 开发者、高级用户（希望拥有个人AI助手） |
| 核心价值 | 数据自主、多渠道统一、Agent原生 |
| 许可证 | MIT开源 |

## 三、核心组件详解

### 1. Gateway 网关

**角色：** 单一长生命周期的守护进程

| 职责 | 说明 |
|------|------|
| 渠道连接 | 维护所有消息渠道的连接（WhatsApp/Telegram/Discord等） |
| 协议服务 | 暴露WebSocket API（请求/响应/事件推送） |
| 路由分发 | 将入站消息路由到正确的Agent |
| 会话管理 | 管理会话状态和历史 |
| 安全认证 | 设备配对、令牌验证、访问控制 |

**关键特性：**
- 一台主机只运行一个Gateway
- 一个WhatsApp会话只被一个Gateway控制
- 所有客户端通过WebSocket连接

---

### 2. Agent 代理

**角色：** 完全隔离的"大脑"

| 组成部分 | 说明 |
|---------|------|
| Workspace | 工作目录（文件、配置） |
| agentDir | 状态目录（认证、会话） |
| Sessions | 会话存储（聊天历史） |

**隔离性：**
- 每个Agent有独立的工作空间
- 每个Agent有独立的认证配置
- 每个Agent有独立的会话存储

**路由规则（优先级从高到低）：**

```
1. peer match（精确DM/群组ID）
2. parentPeer match（线程继承）
3. guildId + roles（Discord角色路由）
4. guildId（Discord服务器）
5. teamId（Slack团队）
6. accountId match（账号匹配）
7. channel-level match（渠道级别）
8. fallback到默认Agent
```

---

### 3. Channel 渠道

**内置渠道：**

| 渠道 | 底层库 | 说明 |
|------|--------|------|
| WhatsApp | Baileys | 最常用的渠道 |
| Telegram | grammy | Bot API |
| Discord | discord.js | Bot + Message Content Intent |
| iMessage | BlueBubbles | macOS配合 |
| Signal | signal-cli | 隐私优先 |
| Slack | Slack API | 工作场景 |
| DingTalk | 企业内部API | 国内场景 |

**插件渠道：**
- Mattermost、Matrix、Microsoft Teams、Nostr等

## 四、数据流

### 消息入站流程

```
用户发送消息
      │
      ▼
Channel接收（如WhatsApp）
      │
      ▼
Gateway验证（配对/允许列表）
      │
      ▼
路由引擎匹配
   ┌────┴────┐
   │ 渠道ID  │
   │ 账号ID  │
   │ 发送者  │
   │ 群组ID  │
   └────┬────┘
        │
        ▼
   选择目标Agent
        │
        ▼
Agent Runtime处理
   ┌────┴────┐
   │ 加载上下文 │
   │ 调用模型  │
   │ 执行工具  │
   │ 生成回复  │
   └────┬────┘
        │
        ▼
通过原渠道返回
```

## 五、核心概念

### 1. 会话 (Session)

| 类型 | 说明 |
|------|------|
| direct | 私聊会话，collapse到main session |
| group | 群聊会话，独立session |
| isolated | 隔离会话（cron任务等） |

**Session ID格式：** `agent:<agentId>:<sessionKey>`

### 2. 绑定 (Binding)

**作用：** 将入站消息路由到特定Agent

```json5
{
  bindings: [
    { 
      agentId: "main", 
      match: { 
        channel: "whatsapp", 
        accountId: "personal" 
      } 
    },
    { 
      agentId: "work", 
      match: { 
        channel: "whatsapp", 
        accountId: "biz" 
      } 
    },
  ]
}
```

### 3. 配对 (Pairing)

**设备配对流程：**

```
新设备连接 → 发送设备身份 → Gateway检查是否已配对
                                      │
                                   ┌──┴──┐
                                   │已配对│ → 验证设备令牌 → 允许连接
                                   └──┬──┘
                                      │未配对
                                      ▼
                                 本地连接？→ 自动批准
                                      │
                                      │非本地
                                      ▼
                                  需人工批准
                                      │
                                      ▼
                                 颁发设备令牌
```

## 八、扩展机制

### Skills 技能

**加载顺序（后加载覆盖先加载）：**
1. Bundled（内置）
2. Managed（~/.openclaw/skills）
3. Workspace（工作空间/skills）

### Plugins 插件

**作用：** 扩展渠道和功能

```bash
openclaw plugins install mattermost
```

### MCP (Model Context Protocol)

**作用：** 连接外部工具服务器

```bash
openclaw mcp add-server my-server
```

### ACP Harness

**作用：** 让IDE通过ACP协议驱动Agent

```bash
openclaw acp
```

## 九、总结

### 核心设计原则

| 原则 | 说明 |
|------|------|
| 单一网关 | 一个Gateway控制所有渠道 |
| Agent隔离 | 每个Agent独立工作空间和会话 |
| 确定性路由 | 明确的优先级规则 |
| 本地优先 | 自托管、数据自主 |
| 可扩展 | Skills、Plugins、MCP |

### 适用场景

| 场景 | 说明 |
|------|------|
| 个人AI助手 | 从任何消息应用访问 |
| 团队协作 | 多Agent、多账号 |
| 自动化任务 | Cron、Webhook、Hooks |
| 开发测试 | CLI、Web UI调试 |
| 设备控制 | iOS/Android节点 |
